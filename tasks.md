## Процесс

- **Kanban** — GitHub Projects, колонки: `Backlog → In Progress → Review → Done`.
- **WIP-лимит**: не более 2 задач на человека одновременно.
- **Еженедельный синк**: 30 мин, каждый показывает работу, обсуждаем блокеры.
- **Git flow**: feature branch → Pull Request → review (минимум 1 апрув) → merge в `main`.
- Каждый этап заканчивается **работающей демо-версией**.

## Этап 1 (неделя 1–2): «Можно зайти и увидеть сервер»

**Цель:** `docker compose up` поднимает всё. Пользователь может зарегистрироваться, залогиниться, создать сервер и каналы. Фронтенд отображает layout.

**Критерий приёмки:** запустить → открыть браузер → зарегистрироваться → создать сервер → увидеть его в интерфейсе.

## Этап 2 (неделя 3–4): «Можно переписываться с шифрованием»

**Цель:** WebSocket-соединение. Отправка и получение сообщений в реальном времени с E2EE. В базе — blob, в браузере — текст. Видно кто онлайн.

**Критерий приёмки:** два браузера, два пользователя, один сервер → один пишет, другой мгновенно видит → в PostgreSQL — нечитаемый blob.

### Задачи Архитектора (А)

| ID | Задача | Описание | Результат |
|:--|:--|:--|:--|
| A2.1 | Guild Module: роли и RBAC | Модель ролей с bitmask-правами. CRUD ролей. Назначение ролей участникам. Middleware проверки прав. | Можно создать роль, назначить, проверить доступ |
| A2.2 | Guild Module: инвайты | Генерация инвайт-ссылок (с TTL и лимитом использований). Вступление по инвайту. | Инвайт-ссылка → вступление → участник в сервере |
| A2.3 | Redis-кэш прав | При проверке прав — сначала Redis, потом БД. Инвалидация при смене ролей. | Проверка прав < 5 мс (Redis hit) |
| A2.4 | WebSocket: подключение и auth | ASP.NET Core WebSocket endpoint. Авторизация по JWT при handshake. Регистрация подключения. | Браузер устанавливает WS, сервер знает userId |
| A2.5 | WebSocket: broadcast | Рассылка события всем онлайн-участникам канала. Маршрутизация по guild → channel → userId. | Сообщение доходит до всех онлайн в канале |
